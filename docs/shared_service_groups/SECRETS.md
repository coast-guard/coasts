# SSG-Native Secrets

`Coastfile.shared_service_groups` accepts `[secrets.<name>]` blocks alongside `[shared_services.<name>]`. SSG secrets are extracted at `coast ssg build` time, stored encrypted in the daemon keystore under a per-project namespace, and injected into the SSG's inner services at `coast ssg run` time via a generated `compose.override.yml`.

This page documents the SSG-side pipeline. For consumer-side `[secrets.*]` (the credentials your app uses to talk to a shared service), see [Coastfile: Secrets](../coastfiles/SECRETS.md). The two layers are independent -- you can use either, both, or neither. See [Composition](#composition-with-consumer-side-secrets) below for how they interact.

## Schema

The `[secrets.<name>]` shape mirrors the regular Coastfile (same extractors, same `inject` syntax). See [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname) for the full reference.

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

By default, a secret is injected into every declared service. Add `services = [...]` to scope it:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Build Pipeline

`coast ssg build` runs an `Extracting secrets` step between compose synthesis and image pulls. Skipped silently when the Coastfile has no `[secrets]` block. Per declared secret:

1. The extractor registry runs the chosen extractor (`env` / `file` / `command` / `keychain` / 1password / custom `coast-extractor-<name>`).
2. The decrypted value lands in the keystore under `coast_image = "ssg:<project>"`. The `ssg:` prefix is a sentinel namespace that can never collide with a regular Coast keystore entry (project names can't contain `:`).
3. A `BuildProgressEvent::item` event streams out per secret so the SPA renders one row per extractor.
4. Existing `ssg:<project>` rows are deleted before the new extraction starts. **Rebuild semantics:** every `coast ssg build` re-extracts, and the keystore always reflects the most recent build's secret config.

The `manifest.json` carries a `secret_injects` array describing the *shape* (name, inject type, target, services) -- never the values. The build hash also folds in `secrets[].extractor`, `secrets[].inject`, and the per-extractor params, so a Coastfile edit that only changes secret config still produces a fresh `build_id`.

## Run Pipeline

`coast ssg run` runs a `Materializing secrets` step (step 6 of 7) between `Loading cached images` and `Starting inner services`. Skipped silently when `manifest.secret_injects` is empty.

The materialize step:

1. Opens the keystore and reads every row for `ssg:<project>`.
2. For each `manifest.secret_injects` entry, looks up the matching keystore row by name.
3. **`env:`** injects -> appends `{NAME: <decrypted>}` under `services.<svc>.environment` per service in the override.
4. **`file:`** injects -> writes decrypted bytes to `~/.coast/ssg/runs/<project>/secrets/<basename>` with mode `0600`, then appends a read-only volume entry under `services.<svc>.volumes`.
5. Renders a deterministic YAML override:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. Writes the override to `~/.coast/ssg/runs/<project>/compose.override.yml`.

The per-run scratch dir is bind-mounted into the outer DinD at `/coast-runtime/`, alongside the immutable `/coast-artifact/`. The compose argv inside the SSG becomes:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start` re-runs `materialize_secrets`, so the override file is refreshed on every start. A `coast ssg secrets clear` between stop and start takes effect on the next start.

The artifact dir (`/coast-artifact/compose.yml`, manifest, ssg-coastfile) stays read-only and byte-stable across runs -- secrets never touch it. The override layer is the only mutable piece.

### File-secret layout

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # generated each run
  secrets/
    jwt                            # mode 0600, decrypted bytes
    tls_cert                       # mode 0600
```

Inside the SSG's outer DinD, the same directory is visible at `/coast-runtime/`. The compose override mounts each file into the inner service's container at the configured `inject_target`, read-only.

## Lifecycle: keystore is never auto-purged

The keystore is deliberately preserved across container teardown:

| Verb | Containers | Inner volumes | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | removed | preserved | **preserved** |
| `coast ssg rm --with-data` | removed | dropped | **preserved** |
| `coast ssg secrets clear` | unchanged | unchanged | **dropped** |

The reasoning: removing an SSG is about wiping containers and data. Destroying credentials a user laboriously extracted from a keychain or 1Password is a separate, more dangerous operation that should be explicit.

`coast ssg secrets clear` is idempotent. Running it twice in a row reports `Cleared 2`, then `Cleared 0`.

## Doctor

`coast ssg doctor` includes one finding per declared secret:

- **`ok`** -- the manifest declares a secret AND the keystore has a matching row.
- **`info`** -- "Declared in Coastfile but missing from the keystore. Run `coast ssg build` to re-extract or remove the [secrets.<name>] block." Triggered when a manifest declaration lacks a keystore row -- usually because someone ran `coast ssg secrets clear` after the build, or copied an artifact between machines.

These findings are non-fatal; the SSG can run without `info` secrets, but the corresponding env vars / files won't be injected.

## Composition with consumer-side `[secrets.*]`

Both secret layers can be used together freely:

| Layer | Where declared | Where extracted | Where injected |
|---|---|---|---|
| **Consumer-side** | the consumer project's regular `Coastfile` | `coast build` time, into `coast_image = "<consumer_project_image>"` | inside the consumer's coast DinD (env vars in compose, file mounts via `docker exec` materialization) |
| **SSG-side** | `Coastfile.shared_service_groups` | `coast ssg build` time, into `coast_image = "ssg:<project>"` | inside the SSG's outer DinD via the per-run `compose.override.yml` |

There is no namespace overlap (`<project>` vs `ssg:<project>`), no injection-target overlap (the consumer's app containers vs the SSG's inner services), and no shared run-time state. Mental model:

- **Use consumer-side `[secrets.*]`** when your app needs a credential the SSG doesn't (a third-party API key, a JWT signing key your app validates).
- **Use SSG-side `[secrets.*]`** when a shared service needs a credential at startup (`POSTGRES_PASSWORD`, an inner service's TLS cert).

## Why an override file instead of `docker exec`?

The consumer-side `file:` inject pipeline uses `docker exec` to materialize secret bytes inside the consumer's coast DinD before compose-up. The SSG side picks a different mechanism for three reasons:

- Bind-mounting a host scratch dir at `/coast-runtime/` is much cheaper than a privileged exec for every secret.
- `-f` compose layering is a built-in feature; the inner compose stack handles the env/volume merge natively. No custom rewrite logic.
- Keeping the artifact dir (`/coast-artifact/`) read-only is a build-pipeline invariant. The override lives outside it, so build artifacts stay byte-stable.
