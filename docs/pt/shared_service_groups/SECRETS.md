# Segredos Nativos do SSG

`Coastfile.shared_service_groups` aceita blocos `[secrets.<name>]` junto com `[shared_services.<name>]`. Os segredos do SSG são extraídos no momento de `coast ssg build`, armazenados criptografados no keystore do daemon sob um namespace por projeto, e injetados nos serviços internos do SSG no momento de `coast ssg run` por meio de um `compose.override.yml` gerado.

Esta página documenta o pipeline do lado do SSG. Para `[secrets.*]` do lado do consumidor (as credenciais que seu app usa para falar com um serviço compartilhado), veja [Coastfile: Secrets](../coastfiles/SECRETS.md). As duas camadas são independentes -- você pode usar uma, ambas, ou nenhuma. Veja [Composition](#composition-with-consumer-side-secrets) abaixo para como elas interagem.

## Esquema

O formato de `[secrets.<name>]` espelha o Coastfile regular (mesmos extratores, mesma sintaxe de `inject`). Veja [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname) para a referência completa.

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

Por padrão, um segredo é injetado em todo serviço declarado. Adicione `services = [...]` para restringi-lo:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Pipeline de Build

`coast ssg build` executa uma etapa `Extracting secrets` entre a síntese do compose e os pulls de imagem. Ignorada silenciosamente quando o Coastfile não tem bloco `[secrets]`. Para cada segredo declarado:

1. O registro de extratores executa o extrator escolhido (`env` / `file` / `command` / `keychain` / 1password / `coast-extractor-<name>` personalizado).
2. O valor descriptografado vai para o keystore sob `coast_image = "ssg:<project>"`. O prefixo `ssg:` é um namespace sentinela que nunca pode colidir com uma entrada regular do keystore do Coast (nomes de projeto não podem conter `:`).
3. Um evento `BuildProgressEvent::item` é emitido por segredo para que o SPA renderize uma linha por extrator.
4. Linhas existentes de `ssg:<project>` são excluídas antes de a nova extração começar. **Semântica de rebuild:** todo `coast ssg build` reextrai, e o keystore sempre reflete a configuração de segredos do build mais recente.

O `manifest.json` carrega um array `secret_injects` descrevendo o *formato* (nome, tipo de inject, destino, serviços) -- nunca os valores. O hash do build também incorpora `secrets[].extractor`, `secrets[].inject` e os parâmetros por extrator, de modo que uma edição no Coastfile que apenas altere a configuração de segredos ainda produza um `build_id` novo.

## Pipeline de Execução

`coast ssg run` executa uma etapa `Materializing secrets` (etapa 6 de 7) entre `Loading cached images` e `Starting inner services`. Ignorada silenciosamente quando `manifest.secret_injects` está vazio.

A etapa de materialização:

1. Abre o keystore e lê toda linha de `ssg:<project>`.
2. Para cada entrada em `manifest.secret_injects`, procura a linha correspondente no keystore pelo nome.
3. Injeções **`env:`** -> anexa `{NAME: <decrypted>}` em `services.<svc>.environment` por serviço no override.
4. Injeções **`file:`** -> grava bytes descriptografados em `~/.coast/ssg/runs/<project>/secrets/<basename>` com modo `0600`, depois anexa uma entrada de volume somente leitura em `services.<svc>.volumes`.
5. Renderiza um override YAML determinístico:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. Grava o override em `~/.coast/ssg/runs/<project>/compose.override.yml`.

O diretório temporário por execução é montado por bind no DinD externo em `/coast-runtime/`, ao lado do imutável `/coast-artifact/`. O argv do compose dentro do SSG se torna:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start` executa `materialize_secrets` novamente, então o arquivo de override é atualizado a cada inicialização. Um `coast ssg secrets clear` entre stop e start entra em efeito na próxima inicialização.

O diretório de artifact (`/coast-artifact/compose.yml`, manifest, ssg-coastfile) permanece somente leitura e estável em bytes entre execuções -- segredos nunca o tocam. A camada de override é a única parte mutável.

### Layout de segredos em arquivo

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # gerado a cada execução
  secrets/
    jwt                            # modo 0600, bytes descriptografados
    tls_cert                       # modo 0600
```

Dentro do DinD externo do SSG, o mesmo diretório é visível em `/coast-runtime/`. O compose override monta cada arquivo no container do serviço interno no `inject_target` configurado, em modo somente leitura.

## Ciclo de vida: o keystore nunca é purgado automaticamente

O keystore é deliberadamente preservado após a desmontagem dos containers:

| Verb | Containers | Volumes internos | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | removidos | preservados | **preservado** |
| `coast ssg rm --with-data` | removidos | descartados | **preservado** |
| `coast ssg secrets clear` | inalterados | inalterados | **descartado** |

A razão: remover um SSG diz respeito a apagar containers e dados. Destruir credenciais que um usuário extraiu com esforço de um keychain ou do 1Password é uma operação separada, mais perigosa, e deve ser explícita.

`coast ssg secrets clear` é idempotente. Executá-lo duas vezes seguidas reporta `Cleared 2`, depois `Cleared 0`.

## Doctor

`coast ssg doctor` inclui uma verificação por segredo declarado:

- **`ok`** -- o manifest declara um segredo E o keystore tem uma linha correspondente.
- **`info`** -- "Declarado no Coastfile mas ausente do keystore. Execute `coast ssg build` para reextrair ou remova o bloco [secrets.<name>]." Disparado quando uma declaração no manifest não tem linha correspondente no keystore -- geralmente porque alguém executou `coast ssg secrets clear` após o build, ou copiou um artifact entre máquinas.

Essas verificações não são fatais; o SSG pode executar sem segredos `info`, mas as variáveis de ambiente / arquivos correspondentes não serão injetados.

## Composição com `[secrets.*]` do lado do consumidor

Ambas as camadas de segredos podem ser usadas juntas livremente:

| Layer | Onde declarado | Onde extraído | Onde injetado |
|---|---|---|---|
| **Lado do consumidor** | o `Coastfile` regular do projeto consumidor | no momento de `coast build`, em `coast_image = "<consumer_project_image>"` | dentro do coast DinD do consumidor (variáveis de ambiente no compose, mounts de arquivo via materialização com `docker exec`) |
| **Lado do SSG** | `Coastfile.shared_service_groups` | no momento de `coast ssg build`, em `coast_image = "ssg:<project>"` | dentro do DinD externo do SSG via `compose.override.yml` por execução |

Não há sobreposição de namespace (`<project>` vs `ssg:<project>`), nem sobreposição de alvo de injeção (os containers do app do consumidor vs os serviços internos do SSG), e nem estado compartilhado em tempo de execução. Modelo mental:

- **Use `[secrets.*]` do lado do consumidor** quando seu app precisar de uma credencial que o SSG não precisa (uma chave de API de terceiros, uma chave de assinatura JWT que seu app valida).
- **Use `[secrets.*]` do lado do SSG** quando um serviço compartilhado precisar de uma credencial na inicialização (`POSTGRES_PASSWORD`, o certificado TLS de um serviço interno).

## Por que um arquivo de override em vez de `docker exec`?

O pipeline de inject `file:` do lado do consumidor usa `docker exec` para materializar bytes do segredo dentro do coast DinD do consumidor antes do compose-up. O lado do SSG escolhe um mecanismo diferente por três razões:

- Fazer bind-mount de um diretório temporário do host em `/coast-runtime/` é muito mais barato do que um exec privilegiado para cada segredo.
- A sobreposição de compose com `-f` é um recurso nativo; a stack compose interna lida com a mesclagem de env/volume nativamente. Sem lógica personalizada de reescrita.
- Manter o diretório de artifact (`/coast-artifact/`) como somente leitura é uma invariante do pipeline de build. O override vive fora dele, então os artifacts de build permanecem estáveis em bytes.
