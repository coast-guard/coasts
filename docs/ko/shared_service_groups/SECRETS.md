# SSG-Native Secrets

`Coastfile.shared_service_groups`는 `[shared_services.<name>]` 블록과 함께 `[secrets.<name>]` 블록도 허용합니다. SSG 시크릿은 `coast ssg build` 시점에 추출되어, 프로젝트별 네임스페이스 아래 데몬 키스토어에 암호화되어 저장되며, `coast ssg run` 시점에 생성된 `compose.override.yml`을 통해 SSG의 내부 서비스들에 주입됩니다.

이 페이지는 SSG 측 파이프라인을 문서화합니다. 소비자 측 `[secrets.*]`(앱이 공유 서비스와 통신할 때 사용하는 자격 증명)에 대해서는 [Coastfile: Secrets](../coastfiles/SECRETS.md)를 참조하세요. 두 계층은 서로 독립적입니다 -- 둘 중 하나만 사용해도 되고, 둘 다 사용해도 되며, 둘 다 사용하지 않아도 됩니다. 상호 작용 방식은 아래의 [Composition](#composition-with-consumer-side-secrets)을 참조하세요.

## Schema

`[secrets.<name>]` 형태는 일반 Coastfile을 그대로 따릅니다(동일한 extractor, 동일한 `inject` 문법). 전체 참조는 [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname)를 참조하세요.

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

기본적으로 시크릿은 선언된 모든 서비스에 주입됩니다. 범위를 제한하려면 `services = [...]`를 추가하세요:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Build Pipeline

`coast ssg build`는 compose 생성과 이미지 pull 사이에서 `Extracting secrets` 단계를 실행합니다. Coastfile에 `[secrets]` 블록이 없으면 조용히 건너뜁니다. 선언된 각 시크릿마다 다음을 수행합니다:

1. extractor registry가 선택된 extractor를 실행합니다 (`env` / `file` / `command` / `keychain` / 1password / 사용자 정의 `coast-extractor-<name>`).
2. 복호화된 값은 `coast_image = "ssg:<project>"` 아래의 키스토어에 저장됩니다. `ssg:` 접두사는 일반 Coast 키스토어 항목과 절대 충돌할 수 없는 센티널 네임스페이스입니다(프로젝트 이름에는 `:`가 포함될 수 없습니다).
3. 시크릿마다 `BuildProgressEvent::item` 이벤트 하나가 스트리밍되어 SPA가 extractor별로 한 줄씩 렌더링합니다.
4. 새 추출이 시작되기 전에 기존 `ssg:<project>` 행은 삭제됩니다. **재빌드 의미론:** 모든 `coast ssg build`는 다시 추출을 수행하며, 키스토어는 항상 가장 최근 빌드의 시크릿 설정을 반영합니다.

`manifest.json`에는 *형태*를 설명하는 `secret_injects` 배열(이름, inject 타입, 대상, 서비스들)이 포함되지만, 값 자체는 절대 포함되지 않습니다. 빌드 해시는 `secrets[].extractor`, `secrets[].inject`, 그리고 extractor별 파라미터도 함께 반영하므로, 시크릿 설정만 변경하는 Coastfile 수정도 새로운 `build_id`를 생성합니다.

## Run Pipeline

`coast ssg run`은 `Loading cached images`와 `Starting inner services` 사이에서 `Materializing secrets` 단계(7단계 중 6단계)를 실행합니다. `manifest.secret_injects`가 비어 있으면 조용히 건너뜁니다.

materialize 단계는 다음을 수행합니다:

1. 키스토어를 열고 `ssg:<project>`에 대한 모든 행을 읽습니다.
2. 각 `manifest.secret_injects` 항목에 대해, 이름으로 일치하는 키스토어 행을 조회합니다.
3. **`env:`** 주입 -> override에서 서비스별 `services.<svc>.environment` 아래에 `{NAME: <decrypted>}`를 추가합니다.
4. **`file:`** 주입 -> 복호화된 바이트를 `~/.coast/ssg/runs/<project>/secrets/<basename>`에 모드 `0600`으로 기록한 뒤, `services.<svc>.volumes` 아래에 읽기 전용 볼륨 항목을 추가합니다.
5. 결정적인 YAML override를 렌더링합니다:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. override를 `~/.coast/ssg/runs/<project>/compose.override.yml`에 기록합니다.

실행별 scratch 디렉터리는 변경 불가능한 `/coast-artifact/`와 함께 외부 DinD에 `/coast-runtime/`으로 bind-mount됩니다. SSG 내부의 compose argv는 다음과 같습니다:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start`는 `materialize_secrets`를 다시 실행하므로, override 파일은 시작할 때마다 새로 갱신됩니다. stop과 start 사이에 `coast ssg secrets clear`를 실행하면 다음 start에 반영됩니다.

artifact 디렉터리(`/coast-artifact/compose.yml`, manifest, ssg-coastfile)는 실행 간 읽기 전용이며 바이트 단위로 안정적으로 유지됩니다 -- 시크릿은 절대 여기에 닿지 않습니다. override 레이어만이 유일한 변경 가능 요소입니다.

### File-secret layout

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # 각 run마다 생성됨
  secrets/
    jwt                            # mode 0600, 복호화된 바이트
    tls_cert                       # mode 0600
```

SSG의 외부 DinD 내부에서는 동일한 디렉터리가 `/coast-runtime/`에 보입니다. compose override는 각 파일을 설정된 `inject_target`에 따라 내부 서비스 컨테이너 안으로 읽기 전용으로 마운트합니다.

## Lifecycle: keystore is never auto-purged

키스토어는 컨테이너 teardown 이후에도 의도적으로 유지됩니다:

| Verb | Containers | Inner volumes | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | 제거됨 | 유지됨 | **유지됨** |
| `coast ssg rm --with-data` | 제거됨 | 삭제됨 | **유지됨** |
| `coast ssg secrets clear` | 변경 없음 | 변경 없음 | **삭제됨** |

이유는 다음과 같습니다: SSG를 제거하는 것은 컨테이너와 데이터를 지우는 작업입니다. 사용자가 힘들게 keychain이나 1Password에서 추출한 자격 증명을 파괴하는 것은 별개의, 더 위험한 작업이므로 명시적으로 수행되어야 합니다.

`coast ssg secrets clear`는 멱등적입니다. 연속으로 두 번 실행하면 `Cleared 2`, 그다음 `Cleared 0`이 보고됩니다.

## Doctor

`coast ssg doctor`는 선언된 각 시크릿에 대해 하나의 진단 결과를 포함합니다:

- **`ok`** -- manifest가 시크릿을 선언하고 있으며, 키스토어에도 일치하는 행이 있습니다.
- **`info`** -- "Coastfile에는 선언되어 있지만 키스토어에는 없습니다. 다시 추출하려면 `coast ssg build`를 실행하거나 [secrets.<name>] 블록을 제거하세요." manifest 선언에 대응하는 키스토어 행이 없을 때 발생합니다 -- 보통 빌드 후 누군가 `coast ssg secrets clear`를 실행했거나, 머신 간에 artifact를 복사한 경우입니다.

이 진단 결과들은 치명적이지 않습니다; `info` 시크릿이 없어도 SSG는 실행될 수 있지만, 해당 env 변수 / 파일은 주입되지 않습니다.

## Composition with consumer-side `[secrets.*]`

두 시크릿 계층은 함께 자유롭게 사용할 수 있습니다:

| Layer | Where declared | Where extracted | Where injected |
|---|---|---|---|
| **Consumer-side** | 소비자 프로젝트의 일반 `Coastfile` | `coast build` 시점, `coast_image = "<consumer_project_image>"` 안으로 | 소비자의 coast DinD 내부(compose의 env vars, `docker exec` materialization을 통한 파일 마운트) |
| **SSG-side** | `Coastfile.shared_service_groups` | `coast ssg build` 시점, `coast_image = "ssg:<project>"` 안으로 | 실행별 `compose.override.yml`을 통해 SSG의 외부 DinD 내부로 |

네임스페이스 중복도 없고(`ssg:<project>` vs `<project>`), 주입 대상 중복도 없으며(소비자 앱 컨테이너 vs SSG 내부 서비스), 공유되는 런타임 상태도 없습니다. 개념적으로는 다음과 같습니다:

- 앱에 SSG에는 필요 없는 자격 증명(서드파티 API 키, 앱이 검증하는 JWT 서명 키)이 필요하면 **consumer-side `[secrets.*]`**를 사용하세요.
- 공유 서비스가 시작 시점에 자격 증명(`POSTGRES_PASSWORD`, 내부 서비스의 TLS 인증서)을 필요로 하면 **SSG-side `[secrets.*]`**를 사용하세요.

## Why an override file instead of `docker exec`?

consumer-side `file:` 주입 파이프라인은 compose-up 전에 소비자의 coast DinD 내부에서 시크릿 바이트를 materialize하기 위해 `docker exec`를 사용합니다. SSG 측은 다음 세 가지 이유로 다른 메커니즘을 선택합니다:

- 호스트 scratch 디렉터리를 `/coast-runtime/`에 bind-mount하는 것이 시크릿마다 privileged exec를 수행하는 것보다 훨씬 저렴합니다.
- `-f` compose 레이어링은 내장 기능입니다; 내부 compose 스택이 env/volume 병합을 기본적으로 처리합니다. 사용자 정의 rewrite 로직이 필요 없습니다.
- artifact 디렉터리(`/coast-artifact/`)를 읽기 전용으로 유지하는 것은 빌드 파이프라인의 불변 조건입니다. override는 그 밖에 존재하므로, 빌드 artifact는 바이트 단위로 안정적으로 유지됩니다.
