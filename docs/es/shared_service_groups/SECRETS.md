# Secretos nativos de SSG

`Coastfile.shared_service_groups` acepta bloques `[secrets.<name>]` junto con `[shared_services.<name>]`. Los secretos de SSG se extraen en el momento de `coast ssg build`, se almacenan cifrados en el keystore del daemon bajo un espacio de nombres por proyecto, y se inyectan en los servicios internos del SSG en el momento de `coast ssg run` mediante un `compose.override.yml` generado.

Esta página documenta el pipeline del lado de SSG. Para `[secrets.*]` del lado del consumidor (las credenciales que tu aplicación usa para comunicarse con un servicio compartido), consulta [Coastfile: Secrets](../coastfiles/SECRETS.md). Las dos capas son independientes -- puedes usar una, ambas o ninguna. Consulta [Composición](#composition-with-consumer-side-secrets) a continuación para ver cómo interactúan.

## Esquema

La forma de `[secrets.<name>]` refleja la del Coastfile regular (mismos extractores, misma sintaxis de `inject`). Consulta [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname) para la referencia completa.

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

De forma predeterminada, un secreto se inyecta en cada servicio declarado. Agrega `services = [...]` para delimitar su alcance:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Pipeline de compilación

`coast ssg build` ejecuta un paso `Extracting secrets` entre la síntesis de compose y las descargas de imágenes. Se omite silenciosamente cuando el Coastfile no tiene un bloque `[secrets]`. Por cada secreto declarado:

1. El registro de extractores ejecuta el extractor elegido (`env` / `file` / `command` / `keychain` / 1password / `coast-extractor-<name>` personalizado).
2. El valor descifrado se almacena en el keystore bajo `coast_image = "ssg:<project>"`. El prefijo `ssg:` es un espacio de nombres centinela que nunca puede colisionar con una entrada regular del keystore de Coast (los nombres de proyecto no pueden contener `:`).
3. Se emite un evento `BuildProgressEvent::item` por secreto para que la SPA renderice una fila por extractor.
4. Las filas existentes de `ssg:<project>` se eliminan antes de que comience la nueva extracción. **Semántica de reconstrucción:** cada `coast ssg build` vuelve a extraer, y el keystore siempre refleja la configuración de secretos de la compilación más reciente.

El `manifest.json` incluye un arreglo `secret_injects` que describe la *forma* (nombre, tipo de inyección, destino, servicios) -- nunca los valores. El hash de compilación también incorpora `secrets[].extractor`, `secrets[].inject` y los parámetros por extractor, por lo que una edición del Coastfile que solo cambie la configuración de secretos sigue produciendo un `build_id` nuevo.

## Pipeline de ejecución

`coast ssg run` ejecuta un paso `Materializing secrets` (paso 6 de 7) entre `Loading cached images` y `Starting inner services`. Se omite silenciosamente cuando `manifest.secret_injects` está vacío.

El paso de materialización:

1. Abre el keystore y lee cada fila de `ssg:<project>`.
2. Para cada entrada de `manifest.secret_injects`, busca la fila correspondiente en el keystore por nombre.
3. Inyecciones **`env:`** -> agrega `{NAME: <decrypted>}` en `services.<svc>.environment` por servicio en el override.
4. Inyecciones **`file:`** -> escribe los bytes descifrados en `~/.coast/ssg/runs/<project>/secrets/<basename>` con modo `0600`, luego agrega una entrada de volumen de solo lectura en `services.<svc>.volumes`.
5. Renderiza un override YAML determinista:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. Escribe el override en `~/.coast/ssg/runs/<project>/compose.override.yml`.

El directorio temporal por ejecución se monta mediante bind en el DinD externo en `/coast-runtime/`, junto con el inmutable `/coast-artifact/`. El argv de compose dentro del SSG pasa a ser:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start` vuelve a ejecutar `materialize_secrets`, por lo que el archivo override se actualiza en cada inicio. Un `coast ssg secrets clear` entre stop y start surte efecto en el siguiente inicio.

El directorio de artefactos (`/coast-artifact/compose.yml`, manifest, ssg-coastfile) permanece de solo lectura y estable byte a byte entre ejecuciones -- los secretos nunca lo tocan. La capa override es la única pieza mutable.

### Diseño de secretos de archivo

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # generated each run
  secrets/
    jwt                            # mode 0600, decrypted bytes
    tls_cert                       # mode 0600
```

Dentro del DinD externo del SSG, el mismo directorio es visible en `/coast-runtime/`. El compose override monta cada archivo en el contenedor del servicio interno en el `inject_target` configurado, en modo de solo lectura.

## Ciclo de vida: el keystore nunca se purga automáticamente

El keystore se preserva deliberadamente a través del desmontaje de contenedores:

| Verb | Containers | Inner volumes | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | eliminados | preservados | **preservado** |
| `coast ssg rm --with-data` | eliminados | eliminados | **preservado** |
| `coast ssg secrets clear` | sin cambios | sin cambios | **eliminado** |

El razonamiento: eliminar un SSG consiste en borrar contenedores y datos. Destruir credenciales que un usuario extrajo laboriosamente desde un keychain o 1Password es una operación aparte, más peligrosa, que debe ser explícita.

`coast ssg secrets clear` es idempotente. Ejecutarlo dos veces seguidas informa `Cleared 2`, luego `Cleared 0`.

## Doctor

`coast ssg doctor` incluye un hallazgo por cada secreto declarado:

- **`ok`** -- el manifiesto declara un secreto Y el keystore tiene una fila correspondiente.
- **`info`** -- "Declarado en Coastfile pero ausente del keystore. Ejecuta `coast ssg build` para volver a extraerlo o elimina el bloque [secrets.<name>]." Se activa cuando una declaración del manifiesto no tiene una fila en el keystore -- normalmente porque alguien ejecutó `coast ssg secrets clear` después de la compilación, o copió un artefacto entre máquinas.

Estos hallazgos no son fatales; el SSG puede ejecutarse sin secretos `info`, pero las variables de entorno / archivos correspondientes no se inyectarán.

## Composición con `[secrets.*]` del lado del consumidor

Ambas capas de secretos pueden usarse juntas libremente:

| Layer | Where declared | Where extracted | Where injected |
|---|---|---|---|
| **Consumer-side** | el `Coastfile` regular del proyecto consumidor | en el momento de `coast build`, en `coast_image = "<consumer_project_image>"` | dentro del DinD coast del consumidor (variables de entorno en compose, montajes de archivos mediante materialización con `docker exec`) |
| **SSG-side** | `Coastfile.shared_service_groups` | en el momento de `coast ssg build`, en `coast_image = "ssg:<project>"` | dentro del DinD externo del SSG mediante el `compose.override.yml` por ejecución |

No hay superposición de espacios de nombres (`<project>` vs `ssg:<project>`), no hay superposición de destinos de inyección (los contenedores de la app del consumidor vs los servicios internos del SSG), y no hay estado compartido en tiempo de ejecución. Modelo mental:

- **Usa `[secrets.*]` del lado del consumidor** cuando tu aplicación necesita una credencial que el SSG no necesita (una clave de API de terceros, una clave de firma JWT que tu app valida).
- **Usa `[secrets.*]` del lado del SSG** cuando un servicio compartido necesita una credencial al iniciarse (`POSTGRES_PASSWORD`, el certificado TLS de un servicio interno).

## ¿Por qué un archivo override en lugar de `docker exec`?

El pipeline de inyección `file:` del lado del consumidor usa `docker exec` para materializar bytes secretos dentro del DinD coast del consumidor antes de compose-up. El lado de SSG elige un mecanismo diferente por tres razones:

- Montar mediante bind un directorio temporal del host en `/coast-runtime/` es mucho más barato que un exec privilegiado por cada secreto.
- La superposición de compose con `-f` es una característica integrada; la pila compose interna maneja de forma nativa la fusión de env/volume. No se necesita lógica de reescritura personalizada.
- Mantener el directorio de artefactos (`/coast-artifact/`) en solo lectura es un invariante del pipeline de compilación. El override vive fuera de él, por lo que los artefactos de compilación permanecen estables byte a byte.
