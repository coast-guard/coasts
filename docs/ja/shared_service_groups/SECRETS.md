# SSG-Native Secrets

`Coastfile.shared_service_groups` は、`[shared_services.<name>]` と並んで `[secrets.<name>]` ブロックを受け付けます。SSG シークレットは `coast ssg build` 時に抽出され、プロジェクトごとの名前空間のもとでデーモンのキーストアに暗号化して保存され、`coast ssg run` 時に生成された `compose.override.yml` を介して SSG の内部サービスに注入されます。

このページでは SSG 側のパイプラインについて説明します。consumer 側の `[secrets.*]`（あなたのアプリが共有サービスと通信するために使う認証情報）については、[Coastfile: Secrets](../coastfiles/SECRETS.md) を参照してください。これら 2 つのレイヤーは独立しています -- どちらか一方だけでも、両方でも、どちらも使わなくても構いません。相互作用については、下記の [Composition](#composition-with-consumer-side-secrets) を参照してください。

## Schema

`[secrets.<name>]` の形は通常の Coastfile を踏襲しています（同じ extractor、同じ `inject` 構文）。完全なリファレンスについては [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname) を参照してください。

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

デフォルトでは、シークレットは宣言されたすべてのサービスに注入されます。スコープを絞るには `services = [...]` を追加してください:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Build Pipeline

`coast ssg build` は、compose の合成とイメージ pull の間に `Extracting secrets` ステップを実行します。Coastfile に `[secrets]` ブロックがない場合は何も出さずにスキップされます。宣言された各シークレットごとに:

1. extractor レジストリが選択された extractor（`env` / `file` / `command` / `keychain` / 1password / カスタム `coast-extractor-<name>`）を実行します。
2. 復号された値は、`coast_image = "ssg:<project>"` のもとでキーストアに格納されます。`ssg:` プレフィックスは、通常の Coast キーストアエントリと決して衝突しない番兵用の名前空間です（プロジェクト名には `:` を含められません）。
3. シークレットごとに `BuildProgressEvent::item` イベントがストリームされるため、SPA は extractor ごとに 1 行ずつ描画します。
4. 既存の `ssg:<project>` 行は、新しい抽出開始前に削除されます。**リビルドの意味論:** `coast ssg build` が実行されるたびに再抽出され、キーストアには常に最新のビルドのシークレット設定が反映されます。

`manifest.json` には、*形*（name、inject type、target、services）を記述する `secret_injects` 配列が含まれます -- 値そのものは決して含まれません。また build hash には `secrets[].extractor`、`secrets[].inject`、および extractor ごとのパラメータも織り込まれるため、シークレット設定だけを変更した Coastfile 編集であっても新しい `build_id` が生成されます。

## Run Pipeline

`coast ssg run` は、`Loading cached images` と `Starting inner services` の間で `Materializing secrets` ステップ（7 ステップ中 6 番目）を実行します。`manifest.secret_injects` が空の場合は何も出さずにスキップされます。

materialize ステップでは次を行います:

1. キーストアを開き、`ssg:<project>` のすべての行を読み込みます。
2. `manifest.secret_injects` の各エントリについて、name により一致するキーストア行を探します。
3. **`env:`** injects -> override 内でサービスごとに `services.<svc>.environment` の下へ `{NAME: <decrypted>}` を追加します。
4. **`file:`** injects -> 復号されたバイト列を `~/.coast/ssg/runs/<project>/secrets/<basename>` にモード `0600` で書き込み、その後サービスごとに `services.<svc>.volumes` の下へ読み取り専用ボリュームエントリを追加します。
5. 決定的な YAML override を描画します:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. override を `~/.coast/ssg/runs/<project>/compose.override.yml` に書き込みます。

実行ごとの scratch ディレクトリは、不変の `/coast-artifact/` と並んで、外側の DinD に `/coast-runtime/` として bind mount されます。SSG 内部の compose argv は次のようになります:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start` は `materialize_secrets` を再実行するため、override ファイルは起動のたびに更新されます。stop と start の間に `coast ssg secrets clear` を実行すると、その効果は次の start で反映されます。

artifact ディレクトリ（`/coast-artifact/compose.yml`、manifest、ssg-coastfile）は実行をまたいでも読み取り専用かつバイト単位で安定したままです -- シークレットがそこに触れることはありません。override レイヤーだけが唯一の可変部分です。

### File-secret layout

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # generated each run
  secrets/
    jwt                            # mode 0600, decrypted bytes
    tls_cert                       # mode 0600
```

SSG の外側の DinD 内では、同じディレクトリが `/coast-runtime/` として見えます。compose override は各ファイルを、設定された `inject_target` に従って内部サービスのコンテナ内へ読み取り専用で mount します。

## Lifecycle: keystore is never auto-purged

キーストアは、コンテナの teardown をまたいでも意図的に保持されます:

| Verb | Containers | Inner volumes | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | 削除される | 保持される | **保持される** |
| `coast ssg rm --with-data` | 削除される | 削除される | **保持される** |
| `coast ssg secrets clear` | 変更なし | 変更なし | **削除される** |

その理由: SSG の削除はコンテナとデータを消去するためのものです。ユーザーが keychain や 1Password から苦労して抽出した認証情報を破壊するのは別の、より危険な操作であり、明示的であるべきです。

`coast ssg secrets clear` は冪等です。これを 2 回連続で実行すると、`Cleared 2`、次に `Cleared 0` と報告されます。

## Doctor

`coast ssg doctor` には、宣言された各シークレットごとに 1 つの finding が含まれます:

- **`ok`** -- manifest がシークレットを宣言しており、かつキーストアに一致する行が存在する。
- **`info`** -- "Coastfile では宣言されているが、キーストアには存在しません。再抽出するには `coast ssg build` を実行するか、[secrets.<name>] ブロックを削除してください。" manifest 上の宣言に対応するキーストア行がない場合に発生します -- 通常は、ビルド後に誰かが `coast ssg secrets clear` を実行したか、artifact をマシン間でコピーした場合です。

これらの finding は致命的ではありません。SSG は `info` シークレットなしでも実行できますが、対応する env vars / files は注入されません。

## Composition with consumer-side `[secrets.*]`

両方のシークレットレイヤーは自由に組み合わせて使えます:

| Layer | Where declared | Where extracted | Where injected |
|---|---|---|---|
| **Consumer-side** | consumer プロジェクトの通常の `Coastfile` | `coast build` 時、`coast_image = "<consumer_project_image>"` へ | consumer の coast DinD 内（compose 内の env vars、`docker exec` materialization による file mount） |
| **SSG-side** | `Coastfile.shared_service_groups` | `coast ssg build` 時、`coast_image = "ssg:<project>"` へ | 実行ごとの `compose.override.yml` を介して SSG の外側の DinD 内へ |

名前空間の重なりはありません（`<project>` vs `ssg:<project>`）、注入先ターゲットの重なりもありません（consumer の app コンテナ vs SSG の内部サービス）、また共有される run-time state もありません。メンタルモデルとしては:

- **consumer-side の `[secrets.*]` を使う** のは、SSG には不要だがアプリには必要な認証情報がある場合です（サードパーティ API キー、アプリが検証する JWT 署名鍵など）。
- **SSG-side の `[secrets.*]` を使う** のは、共有サービスが起動時に認証情報を必要とする場合です（`POSTGRES_PASSWORD`、内部サービスの TLS 証明書など）。

## Why an override file instead of `docker exec`?

consumer-side の `file:` inject パイプラインでは、compose-up 前に consumer の coast DinD 内でシークレットのバイト列を materialize するために `docker exec` を使います。SSG 側が別の仕組みを選ぶ理由は 3 つあります:

- `/coast-runtime/` にホストの scratch ディレクトリを bind mount する方が、シークレットごとに privileged exec を行うよりはるかに低コストです。
- `-f` compose layering は組み込み機能です。内部 compose スタックが env/volume のマージをネイティブに処理します。カスタムの rewrite ロジックは不要です。
- artifact ディレクトリ（`/coast-artifact/`）を読み取り専用に保つことは build パイプラインの不変条件です。override はその外側にあるため、build artifact はバイト単位で安定したままです。
