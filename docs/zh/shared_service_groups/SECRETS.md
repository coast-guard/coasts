# SSG 原生 Secrets

`Coastfile.shared_service_groups` 接受 `[secrets.<name>]` 块，与 `[shared_services.<name>]` 并列。SSG secrets 会在 `coast ssg build` 时被提取，以加密形式存储在守护进程 keystore 中、位于按项目划分的命名空间下，并在 `coast ssg run` 时通过生成的 `compose.override.yml` 注入到 SSG 的内部服务中。

本页记录的是 SSG 侧管线。关于消费方侧的 `[secrets.*]`（你的应用用于与共享服务通信的凭证），请参见 [Coastfile: Secrets](../coastfiles/SECRETS.md)。这两层是相互独立的 —— 你可以只用其中一层、两层都用，或者两层都不用。关于它们如何交互，请参见下方的 [Composition](#composition-with-consumer-side-secrets)。

## Schema

`[secrets.<name>]` 的结构与常规 Coastfile 镜像一致（相同的提取器、相同的 `inject` 语法）。完整参考请参见 [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname)。

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

默认情况下，一个 secret 会被注入到每个已声明的服务中。添加 `services = [...]` 可将其限定到特定服务:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Build Pipeline

`coast ssg build` 会在 compose 合成与镜像拉取之间运行一个 `Extracting secrets` 步骤。如果 Coastfile 没有 `[secrets]` 块，则会静默跳过。对于每个已声明的 secret:

1. 提取器注册表运行所选提取器（`env` / `file` / `command` / `keychain` / 1password / 自定义 `coast-extractor-<name>`）。
2. 解密后的值会落入 keystore，位于 `coast_image = "ssg:<project>"` 下。`ssg:` 前缀是一个哨兵命名空间，绝不会与常规 Coast keystore 条目冲突（项目名不能包含 `:`）。
3. 每个 secret 都会流式发送一个 `BuildProgressEvent::item` 事件，因此 SPA 会为每个提取器渲染一行。
4. 在开始新的提取前，现有的 `ssg:<project>` 行会被删除。**重建语义:** 每次 `coast ssg build` 都会重新提取，而 keystore 始终反映最近一次构建的 secret 配置。

`manifest.json` 带有一个 `secret_injects` 数组，用于描述其*形状*（名称、inject 类型、目标、服务）—— 绝不包含值。构建哈希还会纳入 `secrets[].extractor`、`secrets[].inject` 以及每个提取器的参数，因此即使 Coastfile 编辑仅更改了 secret 配置，也仍会生成一个新的 `build_id`。

## Run Pipeline

`coast ssg run` 会在 `Loading cached images` 与 `Starting inner services` 之间运行一个 `Materializing secrets` 步骤（7 步中的第 6 步）。如果 `manifest.secret_injects` 为空，则会静默跳过。

materialize 步骤会:

1. 打开 keystore 并读取 `ssg:<project>` 的所有行。
2. 对于每个 `manifest.secret_injects` 条目，按名称查找匹配的 keystore 行。
3. **`env:`** 注入 -> 在 override 中按服务向 `services.<svc>.environment` 追加 `{NAME: <decrypted>}`。
4. **`file:`** 注入 -> 将解密后的字节写入 `~/.coast/ssg/runs/<project>/secrets/<basename>`，权限模式为 `0600`，然后在 `services.<svc>.volumes` 下追加一个只读卷条目。
5. 渲染一个确定性的 YAML override:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. 将 override 写入 `~/.coast/ssg/runs/<project>/compose.override.yml`。

每次运行的 scratch 目录会被 bind-mount 到外层 DinD 中的 `/coast-runtime/`，与不可变的 `/coast-artifact/` 并列。SSG 内部的 compose argv 变为:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start` 会重新运行 `materialize_secrets`，因此 override 文件会在每次启动时刷新。在 stop 与 start 之间执行 `coast ssg secrets clear`，会在下一次 start 时生效。

artifact 目录（`/coast-artifact/compose.yml`、manifest、ssg-coastfile）在多次运行之间保持只读且字节稳定 —— secrets 永远不会接触它。override 层是唯一可变的部分。

### File-secret layout

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # 每次运行生成
  secrets/
    jwt                            # 模式 0600，解密后的字节
    tls_cert                       # 模式 0600
```

在 SSG 的外层 DinD 内部，同一目录可见于 `/coast-runtime/`。compose override 会将每个文件以只读方式挂载到内部服务容器中配置的 `inject_target` 路径。

## Lifecycle: keystore 永不自动清除

keystore 会被刻意保留，即使容器已被拆除:

| Verb | Containers | Inner volumes | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | 已移除 | 已保留 | **已保留** |
| `coast ssg rm --with-data` | 已移除 | 已丢弃 | **已保留** |
| `coast ssg secrets clear` | 不变 | 不变 | **已丢弃** |

原因是:移除一个 SSG 是关于清除容器和数据。销毁用户辛苦从 keychain 或 1Password 中提取出来的凭证，是一个单独且更危险的操作，应该显式执行。

`coast ssg secrets clear` 是幂等的。连续运行两次会分别报告 `Cleared 2`，然后 `Cleared 0`。

## Doctor

`coast ssg doctor` 会为每个已声明的 secret 包含一条检查结果:

- **`ok`** —— manifest 声明了一个 secret，并且 keystore 中有匹配的行。
- **`info`** —— “已在 Coastfile 中声明，但在 keystore 中缺失。运行 `coast ssg build` 以重新提取，或删除 [secrets.<name>] 块。” 当 manifest 中的声明缺少对应的 keystore 行时触发 —— 通常是因为有人在构建后运行了 `coast ssg secrets clear`，或者在机器之间复制了 artifact。

这些结果不是致命的；SSG 即使缺少 `info` secrets 也能运行，但相应的环境变量 / 文件将不会被注入。

## 与消费方侧 `[secrets.*]` 的组合

两层 secret 可以自由地一起使用:

| Layer | Where declared | Where extracted | Where injected |
|---|---|---|---|
| **Consumer-side** | 消费方项目的常规 `Coastfile` | `coast build` 时，进入 `coast_image = "<consumer_project_image>"` | 在消费方的 coast DinD 内部（compose 中的环境变量，通过 `docker exec` materialization 的文件挂载） |
| **SSG-side** | `Coastfile.shared_service_groups` | `coast ssg build` 时，进入 `coast_image = "ssg:<project>"` | 在 SSG 的外层 DinD 内部，通过每次运行生成的 `compose.override.yml` |

不存在命名空间重叠（`<project>` vs `ssg:<project>`），不存在注入目标重叠（消费方的应用容器 vs SSG 的内部服务），也不存在共享的运行时状态。心智模型如下:

- 当你的应用需要某个 SSG 不需要的凭证时，**使用消费方侧 `[secrets.*]`**（第三方 API key、你的应用用于校验的 JWT signing key）。
- 当某个共享服务在启动时需要凭证时，**使用 SSG 侧 `[secrets.*]`**（`POSTGRES_PASSWORD`、内部服务的 TLS cert）。

## 为什么使用 override 文件而不是 `docker exec`？

消费方侧的 `file:` 注入管线会在 compose-up 之前使用 `docker exec` 在消费方的 coast DinD 内部 materialize secret 字节。SSG 侧选择不同机制有三个原因:

- 将宿主机的 scratch 目录 bind-mount 到 `/coast-runtime/`，比为每个 secret 执行一次特权 exec 便宜得多。
- `-f` compose 分层是内建特性；内部 compose 栈会原生处理 env/volume 合并。无需自定义重写逻辑。
- 保持 artifact 目录（`/coast-artifact/`）只读是构建管线不变式。override 位于其外部，因此构建产物保持字节稳定。
