# Оформление SSG на стороне хоста

Consumer Coast'ы достигают сервисов SSG через слой маршрутизации демона (in-DinD socat -> host socat -> dynamic port). Это отлично работает для контейнеров приложений. Но это не помогает вызывающим на стороне хоста -- MCP, ad-hoc сессиям `psql`, инспектору базы данных в вашем редакторе -- которым нужно подключаться к `localhost:5432` так, как будто сервис находится прямо там.

`coast ssg checkout` решает эту задачу. Он запускает host-level socat, который привязывается к каноническому порту хоста (5432 для Postgres, 6379 для Redis, ...) и перенаправляет трафик на стабильный виртуальный порт проекта. Оттуда существующий socat виртуального порта хоста уже доставляет трафик дальше, к текущему опубликованному динамическому порту SSG.

Всё это работает на уровне проекта. `coast ssg checkout --service postgres` разрешается к проекту, которому принадлежит `Coastfile` в текущем рабочем каталоге; если на этой машине у вас два проекта, только один из них может удерживать канонический порт 5432 одновременно.

## Использование

```bash
coast ssg checkout --service postgres     # bind one service
coast ssg checkout --all                  # bind every SSG service
coast ssg uncheckout --service postgres   # tear down one
coast ssg uncheckout --all                # tear down every active checkout
```

После успешного оформления `coast ssg ports` помечает каждый привязанный сервис как `(checked out)`:

```text
  SERVICE              CANONICAL       DYNAMIC         VIRTUAL    STATUS
  postgres             5432            54201           42000      (checked out)
  redis                6379            54202           42001
```

Consumer Coast'ы всегда достигают сервисов SSG через свою цепочку in-DinD socat -> virtual port, независимо от состояния оформления на стороне хоста. Оформление — это исключительно удобство для хоста.

## Двухпрыжковый форвардер

socat оформления **не** указывает напрямую на динамический порт хоста SSG. Он указывает на стабильный виртуальный порт проекта:

```text
host process            -> 127.0.0.1:5432           (checkout socat, listens here)
                        -> 127.0.0.1:42000          (project's virtual port)
                        -> 127.0.0.1:54201          (SSG's current dynamic port)
                        -> <project>-ssg postgres   (inner service)
```

Двухпрыжковая цепочка означает, что socat оформления продолжает работать при пересборках SSG, даже если динамический порт меняется. Обновляется только socat виртуального порта хоста -- socat канонического порта об этом не знает. См. [Routing](ROUTING.md), чтобы понять, как поддерживается слой host socat.

## Вытеснение держателей экземпляров Coast

Когда вы просите SSG оформить канонический порт, этот порт уже может быть занят. Семантика зависит от того, кто его удерживает:

- **Экземпляр Coast, который был явно оформлен.** `coast checkout <instance>` на каком-то Coast ранее сегодня привязал `localhost:5432` к внутреннему Postgres этого Coast. Оформление SSG **вытесняет** его: демон завершает существующий socat, очищает `port_allocations.socat_pid` для этого Coast и вместо него привязывает socat SSG. CLI выводит явное предупреждение:

  ```text
  Warning: displaced coast 'my-app/dev-2' from canonical port 5432.
  SSG checkout complete: postgres on canonical 5432 -> virtual 42000.
  ```

  Вытеснённый Coast **не** привязывается обратно автоматически, когда вы позже выполняете `coast ssg uncheckout`. Его динамический порт по-прежнему работает, но канонический порт остаётся непривязанным, пока вы снова не выполните `coast checkout my-app/dev-2`.

- **Оформление SSG другого проекта.** Если у `filemap-ssg` уже оформлен 5432, а вы пытаетесь оформить 5432 у `cg-ssg`, демон откажет с явным сообщением, указывающим текущего держателя. Сначала выполните uncheckout для 5432 у `filemap-ssg`.

- **Предыдущая строка оформления SSG с мёртвым `socat_pid`.** Устаревшие метаданные от демона, который аварийно завершился, или после цикла stop/start. Новое оформление молча возвращает строку себе.

- **Что-либо ещё** (host Postgres, который вы запустили вручную, другой демон, `nginx` на порту 8080). `coast ssg checkout` завершится с ошибкой:

  ```text
  error: port 5432 is held by a process outside Coast's tracking. Free the port with `lsof -i :5432` / `kill <pid>` and retry.
  ```

  Флага `--force` нет. Молчаливое завершение неизвестного процесса было сочтено слишком опасным.

## Поведение stop / start

`coast ssg stop` завершает активные процессы socat канонических портов, но **сохраняет сами строки оформления** в state DB.

`coast ssg run` / `start` / `restart` проходят по сохранённым строкам и заново запускают fresh socat канонического порта для каждой строки. Канонический порт (5432) остаётся тем же самым; между циклами `run` меняется только динамический порт, и поскольку socat оформления нацелен на **виртуальный** порт (который тоже стабилен), перепривязка механическая.

Если сервис исчезает из пересобранного SSG, его строка оформления удаляется с предупреждением в ответе run:

```text
SSG run complete. Dropped stale checkout for service 'mongo' (no longer in the active SSG build).
```

`coast ssg rm` удаляет все строки `ssg_port_checkouts` для проекта. `rm` по своей природе разрушителен -- вы явно запросили чистый лист.

## Восстановление после перезапуска демона

После неожиданного перезапуска демона (сбой, `coastd restart`, перезагрузка) `restore_running_state` обращается к таблице `ssg_port_checkouts` и заново запускает каждую строку с учётом текущего распределения dynamic / virtual port. Ваш `localhost:5432` остаётся привязанным несмотря на перезапуски демона.

## Когда стоит оформлять

- Вы хотите направить GUI-клиент базы данных на SSG Postgres проекта.
- Вы хотите, чтобы `psql "postgres://coast:coast@localhost:5432/mydb"` работал без предварительного выяснения динамического порта.
- MCP на вашем хосте нужен стабильный канонический endpoint.
- Coastguard хочет проксировать HTTP admin port SSG.

Когда **не** стоит оформлять:

- Для подключения изнутри consumer Coast -- это уже работает через in-DinD socat к virtual port.
- Если вас устраивает вывод `coast ssg ports` и подстановка динамического порта в ваш инструмент.
