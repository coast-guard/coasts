# Использование общей группы сервисов

Потребляющий Coast подключается к сервисам, принадлежащим SSG его проекта, по одному сервису за раз, используя однострочный флаг в `Coastfile` потребителя. Внутри Coast контейнеры приложения по-прежнему видят `postgres:5432`; слой маршрутизации демона перенаправляет этот трафик во внешний DinD `<project>-ssg` проекта через стабильный виртуальный порт.

SSG, на который ссылается `from_group = true`, **всегда является собственной SSG проекта-потребителя**. Межпроектного шаринга нет. Если `[coast].name` потребителя — `cg`, то `from_group = true` разрешается относительно `Coastfile.shared_service_groups` у `cg-ssg`.

## Синтаксис

Добавьте блок `[shared_services.<name>]` с `from_group = true`:

```toml
# Consumer Coastfile
[coast]
name = "my-app"
compose = "./docker-compose.yml"

[shared_services.postgres]
from_group = true

# Optional per-project overrides:
inject = "env:DATABASE_URL"
# auto_create_db = true       # overrides the SSG service's default
```

Ключ TOML (`postgres` в этом примере) должен совпадать с именем сервиса, объявленным в `Coastfile.shared_service_groups` проекта.

## Запрещённые поля

При `from_group = true` следующие поля отклоняются на этапе разбора:

- `image`
- `ports`
- `env`
- `volumes`

Все они находятся на стороне SSG. Если какое-либо из них указано вместе с `from_group = true`, `coast build` завершится с ошибкой:

```text
error: shared service 'postgres' has from_group = true; the following fields are forbidden: image, ports, env, volumes.
```

## Разрешённые переопределения

Для каждого потребителя по-прежнему допустимы два поля:

- `inject` -- переменная окружения или путь к файлу, через которые предоставляется строка подключения. Разные проекты-потребители могут предоставлять одну и ту же сущность под разными именами переменных окружения.
- `auto_create_db` -- должен ли Coast создавать БД для каждого инстанса внутри этого сервиса во время `coast run`. Переопределяет собственное значение `auto_create_db` сервиса SSG.

## Обнаружение конфликтов

Два блока `[shared_services.<name>]` с одинаковым именем в одном Coastfile отклоняются на этапе разбора. Это правило сохраняется.

Блок с `from_group = true`, который ссылается на имя, не объявленное в `Coastfile.shared_service_groups` проекта, завершится ошибкой во время `coast build`:

```text
error: shared service 'postgres' has from_group = true but no service named 'postgres' is declared in Coastfile.shared_service_groups for project 'my-app'.
```

Это проверка на опечатки. Отдельной runtime-проверки на "drift" нет -- несоответствия формы между потребителем и SSG проявляются в проверке во время сборки, а любые дальнейшие несоответствия во время выполнения естественным образом проявляются как ошибка подключения с точки зрения приложения.

## Автозапуск

`coast run` для потребителя автоматически запускает SSG проекта, если она ещё не запущена:

- Сборка SSG существует, контейнер не запущен -> демон выполняет эквивалент `coast ssg start` (или `run`, если контейнер никогда не создавался), под защитой mutex'а SSG проекта.
- Сборка SSG вообще отсутствует -> жёсткая ошибка:

  ```text
  Project 'my-app' references shared service 'postgres' from the Shared Service Group, but no SSG build exists. Run `coast ssg build` in the directory containing your Coastfile.shared_service_groups.
  ```

- SSG уже запущена -> no-op, `coast run` немедленно продолжается.

События прогресса `SsgStarting` и `SsgStarted` отправляются в поток выполнения, чтобы [Coastguard](../concepts_and_terminology/COASTGUARD.md) мог отнести запуск к проекту-потребителю.

## Как работает маршрутизация

Внутри потребляющего Coast контейнер приложения разрешает `postgres:5432` в SSG проекта с помощью трёх частей:

1. **Alias IP + `extra_hosts`** добавляют `postgres -> <docker0 alias IP>` во внутренний compose потребителя, чтобы DNS-запросы для `postgres` успешно разрешались.
2. **In-DinD socat** слушает на `<alias>:5432` и перенаправляет на `host.docker.internal:<virtual_port>`. Виртуальный порт стабилен для `(project, service, container_port)` -- он не меняется при пересборке SSG.
3. **Host socat** на `<virtual_port>` перенаправляет на `127.0.0.1:<dynamic>`, где `<dynamic>` — текущий опубликованный порт контейнера SSG. Host socat обновляется при пересборке SSG; in-DinD socat у потребителя никогда не нужно менять.

Код приложения и DNS в compose не меняются. Миграция проекта с inline Postgres на SSG Postgres — это небольшое изменение Coastfile (убрать `image`/`ports`/`env`, добавить `from_group = true`) плюс пересборка.

Полное пошаговое описание маршрута, концепций портов и мотивации см. в [Routing](ROUTING.md).

## `auto_create_db`

`auto_create_db = true` для сервиса Postgres или MySQL в SSG заставляет демон создавать базу данных `{instance}_{project}` внутри этого сервиса для каждого потребляющего Coast, который выполняется. Имя базы данных совпадает с тем, что создаёт шаблон inline `[shared_services]`, поэтому URL'ы `inject` согласуются с базой данных, которую создаёт `auto_create_db`.

Создание идемпотентно. Повторный запуск `coast run` для инстанса, чья база данных уже существует, ничего не делает. Базовый SQL идентичен inline-пути, поэтому вывод DDL побайтно одинаков независимо от того, какой шаблон использует ваш проект.

Потребитель может переопределить значение `auto_create_db` сервиса SSG:

```toml
# SSG: auto_create_db = true, but this project doesn't want per-instance DBs.
[shared_services.postgres]
from_group = true
auto_create_db = false
```

## `inject`

`inject` предоставляет строку подключения контейнеру приложения. Формат тот же, что и у [Secrets](../coastfiles/SECRETS.md): `"env:NAME"` создаёт переменную окружения, `"file:/path"` записывает файл внутри coast-контейнера потребителя и монтирует его в режиме только для чтения в каждый незаглушённый сервис внутреннего compose.

Разрешённая строка использует каноническое имя сервиса и канонический порт, а не динамический порт хоста. В этом и состоит весь смысл инвариантности -- контейнеры приложения всегда видят `postgres://coast:coast@postgres:5432/{db}` независимо от того, на каком динамическом порту SSG в данный момент публикуется.

Оба варианта `env:NAME` и `file:/path` полностью реализованы.

Этот `inject` — **конвейер секретов на стороне потребителя**: значение вычисляется из канонических метаданных SSG во время `coast build` и внедряется в coast DinD потребителя. Он независим от конвейера **на стороне SSG** `[secrets.*]` (см. [Secrets](SECRETS.md)), который извлекает значения для потребления *собственными* сервисами SSG.

## Удалённые Coast

Удалённый Coast (созданный с помощью `coast assign --remote ...`) подключается к локальной SSG через обратный SSH-туннель. Локальный демон запускает `ssh -N -R <vport>:localhost:<vport>` с удалённой машины обратно на локальный виртуальный порт; внутри удалённого DinD `extra_hosts: postgres: host-gateway` разрешает `postgres` в IP host-gateway удалённой машины, а SSH-туннель размещает локальную SSG по другую сторону на том же номере виртуального порта.

Обе стороны туннеля используют **виртуальный** порт, а не динамический. Это означает, что локальная пересборка SSG никогда не делает удалённый туннель недействительным.

Туннели коалесцируются по `(project, remote_host, service, container_port)` -- несколько инстансов-потребителей одного и того же проекта на одной и той же удалённой машине разделяют один процесс `ssh -R`. Удаление одного потребителя не разрушает туннель; это происходит только при удалении последнего потребителя.

Практические последствия:

- `coast ssg stop` / `rm` отказываются работать, пока удалённый shadow Coast в данный момент использует SSG. Демон выводит список блокирующих shadow, чтобы было понятно, что использует SSG.
- `coast ssg stop --force` (или `rm --force`) сначала разрушает общий `ssh -R`, а затем продолжает работу. Используйте это, если вы согласны с тем, что удалённые потребители потеряют соединение.

Полную архитектуру удалённого туннеля см. в [Routing](ROUTING.md), а более широкую настройку удалённой машины — в [Remote Coasts](../remote_coasts/README.md).
