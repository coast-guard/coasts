# SSG-Native Secrets

`Coastfile.shared_service_groups` принимает блоки `[secrets.<name>]` наряду с `[shared_services.<name>]`. Секреты SSG извлекаются во время `coast ssg build`, сохраняются в зашифрованном виде в хранилище ключей демона в пространстве имён на проект и внедряются во внутренние сервисы SSG во время `coast ssg run` через сгенерированный `compose.override.yml`.

Эта страница документирует конвейер на стороне SSG. Для `[secrets.*]` на стороне потребителя (учётные данные, которые ваше приложение использует для общения с общим сервисом) см. [Coastfile: Secrets](../coastfiles/SECRETS.md). Эти два слоя независимы -- вы можете использовать любой из них, оба или ни один. О том, как они взаимодействуют, см. [Composition](#composition-with-consumer-side-secrets) ниже.

## Schema

Форма `[secrets.<name>]` повторяет обычный Coastfile (те же экстракторы, тот же синтаксис `inject`). Полное описание см. в [Coastfile: Shared Service Groups -> [secrets.<name>]](../coastfiles/SHARED_SERVICE_GROUPS.md#secretsname).

```toml
[shared_services.postgres]
image = "postgres:16-alpine"
ports = [5432]
env = { POSTGRES_USER = "postgres" }

[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"

[secrets.jwt]
extractor = "env"
var = "MY_JWT_VALUE"
inject = "file:/run/secrets/jwt"
```

По умолчанию секрет внедряется в каждый объявленный сервис. Чтобы ограничить область действия, добавьте `services = [...]`:

```toml
[secrets.pg_password]
extractor = "env"
var = "MY_PG_PASSWORD"
inject = "env:POSTGRES_PASSWORD"
services = ["postgres"]
```

## Build Pipeline

`coast ssg build` выполняет шаг `Extracting secrets` между синтезом compose и загрузкой образов. Если в Coastfile нет блока `[secrets]`, шаг тихо пропускается. Для каждого объявленного секрета:

1. Реестр экстракторов запускает выбранный экстрактор (`env` / `file` / `command` / `keychain` / 1password / пользовательский `coast-extractor-<name>`).
2. Расшифрованное значение попадает в хранилище ключей под `coast_image = "ssg:<project>"`. Префикс `ssg:` — это служебное пространство имён, которое никогда не может пересечься с обычной записью хранилища ключей Coast (имена проектов не могут содержать `:`).
3. Для каждого секрета отправляется событие `BuildProgressEvent::item`, чтобы SPA отображала по одной строке на экстрактор.
4. Существующие записи `ssg:<project>` удаляются до начала нового извлечения. **Семантика пересборки:** каждый `coast ssg build` заново извлекает секреты, и хранилище ключей всегда отражает конфигурацию секретов из самой последней сборки.

`manifest.json` содержит массив `secret_injects`, описывающий *форму* (имя, тип внедрения, цель, сервисы) -- но никогда не значения. Хэш сборки также включает `secrets[].extractor`, `secrets[].inject` и параметры конкретного экстрактора, поэтому изменение Coastfile, затрагивающее только конфигурацию секретов, всё равно создаёт новый `build_id`.

## Run Pipeline

`coast ssg run` выполняет шаг `Materializing secrets` (шаг 6 из 7) между `Loading cached images` и `Starting inner services`. Если `manifest.secret_injects` пуст, шаг тихо пропускается.

Шаг материализации:

1. Открывает хранилище ключей и читает все записи для `ssg:<project>`.
2. Для каждой записи в `manifest.secret_injects` ищет соответствующую запись в хранилище ключей по имени.
3. Внедрения **`env:`** -> добавляют `{NAME: <decrypted>}` в `services.<svc>.environment` для каждого сервиса в override-файле.
4. Внедрения **`file:`** -> записывают расшифрованные байты в `~/.coast/ssg/runs/<project>/secrets/<basename>` с режимом `0600`, затем добавляют запись тома только для чтения в `services.<svc>.volumes`.
5. Рендерит детерминированный YAML override:

   ```yaml
   # Generated by coast-ssg at run time. Do not hand-edit.
   services:
     postgres:
       environment:
         POSTGRES_PASSWORD: "<decrypted>"
       volumes:
         - "/coast-runtime/secrets/jwt:/run/secrets/jwt:ro"
   ```

6. Записывает override в `~/.coast/ssg/runs/<project>/compose.override.yml`.

Scratch-каталог на каждый запуск монтируется bind-монтированием во внешний DinD по пути `/coast-runtime/`, рядом с неизменяемым `/coast-artifact/`. Аргументы compose внутри SSG становятся такими:

```
docker compose -f /coast-artifact/compose.yml \
               -f /coast-runtime/compose.override.yml \
               -p <project>-ssg up -d --remove-orphans
```

`coast ssg start` повторно запускает `materialize_secrets`, поэтому override-файл обновляется при каждом старте. Если между stop и start выполнить `coast ssg secrets clear`, это подействует при следующем старте.

Каталог артефактов (`/coast-artifact/compose.yml`, manifest, ssg-coastfile) остаётся доступным только для чтения и побайтно стабильным между запусками -- секреты его никогда не касаются. Слой override — единственная изменяемая часть.

### File-secret layout

```text
~/.coast/ssg/runs/<project>/
  compose.override.yml             # generated each run
  secrets/
    jwt                            # mode 0600, decrypted bytes
    tls_cert                       # mode 0600
```

Внутри внешнего DinD SSG тот же каталог виден по пути `/coast-runtime/`. Override compose монтирует каждый файл во внутренний контейнер сервиса по настроенному `inject_target` в режиме только для чтения.

## Lifecycle: keystore is never auto-purged

Хранилище ключей намеренно сохраняется после удаления контейнеров:

| Verb | Containers | Inner volumes | Keystore (`ssg:<project>`) |
|---|---|---|---|
| `coast ssg rm` | удалены | сохранены | **сохранено** |
| `coast ssg rm --with-data` | удалены | удалены | **сохранено** |
| `coast ssg secrets clear` | без изменений | без изменений | **удалено** |

Причина в следующем: удаление SSG связано с очисткой контейнеров и данных. Уничтожение учётных данных, которые пользователь с трудом извлёк из keychain или 1Password, — это отдельная, более опасная операция, которая должна быть явной.

`coast ssg secrets clear` идемпотентна. Если запустить её два раза подряд, будет выведено `Cleared 2`, затем `Cleared 0`.

## Doctor

`coast ssg doctor` включает по одному заключению на каждый объявленный секрет:

- **`ok`** -- в manifest объявлен секрет И в хранилище ключей есть соответствующая запись.
- **`info`** -- "Объявлен в Coastfile, но отсутствует в хранилище ключей. Выполните `coast ssg build`, чтобы извлечь его заново, или удалите блок [secrets.<name>]." Срабатывает, когда объявление в manifest не имеет соответствующей записи в хранилище ключей -- обычно потому, что кто-то выполнил `coast ssg secrets clear` после сборки или скопировал артефакт между машинами.

Эти заключения не являются фатальными; SSG может работать без секретов со статусом `info`, но соответствующие переменные окружения / файлы не будут внедрены.

## Composition with consumer-side `[secrets.*]`

Оба слоя секретов можно свободно использовать вместе:

| Layer | Where declared | Where extracted | Where injected |
|---|---|---|---|
| **Consumer-side** | обычный `Coastfile` проекта-потребителя | во время `coast build`, в `coast_image = "<consumer_project_image>"` | внутри coast DinD потребителя (переменные окружения в compose, монтирование файлов через материализацию `docker exec`) |
| **SSG-side** | `Coastfile.shared_service_groups` | во время `coast ssg build`, в `coast_image = "ssg:<project>"` | внутри внешнего DinD SSG через `compose.override.yml` на каждый запуск |

Пересечения пространств имён нет (`<project>` vs `ssg:<project>`), пересечения целей внедрения нет (контейнеры приложения потребителя vs внутренние сервисы SSG), и общего состояния времени выполнения нет. Ментальная модель:

- **Используйте `[secrets.*]` на стороне потребителя**, когда вашему приложению нужны учётные данные, которые не нужны SSG (ключ API стороннего сервиса, ключ подписи JWT, который ваше приложение проверяет).
- **Используйте `[secrets.*]` на стороне SSG**, когда общему сервису нужны учётные данные при запуске (`POSTGRES_PASSWORD`, TLS-сертификат внутреннего сервиса).

## Why an override file instead of `docker exec`?

Конвейер внедрения `file:` на стороне потребителя использует `docker exec` для материализации байтов секрета внутри coast DinD потребителя перед compose-up. Сторона SSG выбирает другой механизм по трём причинам:

- Bind-монтирование scratch-каталога хоста в `/coast-runtime/` значительно дешевле, чем привилегированный exec для каждого секрета.
- Слоение compose через `-f` — это встроенная возможность; внутренний стек compose нативно обрабатывает слияние env/volume. Никакой пользовательской логики переписывания.
- Сохранение каталога артефактов (`/coast-artifact/`) в режиме только для чтения — это инвариант конвейера сборки. Override живёт вне него, поэтому артефакты сборки остаются побайтно стабильными.
