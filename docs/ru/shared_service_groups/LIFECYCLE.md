# Жизненный цикл SSG

SSG каждого проекта — это собственный внешний Docker-in-Docker контейнер с именем `<project>-ssg` (например, `cg-ssg`). Команды жизненного цикла нацелены на SSG того проекта, которому принадлежит `Coastfile` в текущем рабочем каталоге (или проекта, указанного через `--working-dir`). Каждая изменяющая команда сериализуется через мьютекс на уровне проекта в демоне, поэтому два одновременных вызова `coast ssg run` / `coast ssg stop` для одного и того же проекта становятся в очередь, а не конфликтуют — но два разных проекта могут изменять свои SSG параллельно.

## Машина состояний

```text
                     coast ssg build           coast ssg run
(no build)   -->  built     -->     created    -->     running
                                                          |
                                                   coast ssg stop
                                                          v
                                                       stopped
                                                          |
                                                  coast ssg start
                                                          v
                                                       running
                                                          |
                                                   coast ssg rm
                                                          v
                                                      (removed)
```

- `coast ssg build` не создает контейнер. Она создает артефакт на диске в `~/.coast/ssg/<project>/builds/<id>/` и (когда объявлен `[secrets.*]`) извлекает значения секретов в keystore.
- `coast ssg run` создает DinD `<project>-ssg`, выделяет динамические порты хоста, материализует все объявленные секреты в `compose.override.yml` для конкретного запуска и поднимает внутренний compose stack.
- `coast ssg stop` останавливает внешний DinD, но сохраняет контейнер, записи динамических портов и виртуальные порты на уровне проекта, чтобы `start` выполнялась быстро.
- `coast ssg start` заново запускает SSG и повторно материализует секреты (так что `coast ssg secrets clear`, выполненная между stop и start, вступает в силу).
- `coast ssg rm` удаляет внешний контейнер DinD. С `--with-data` она также удаляет внутренние именованные тома (содержимое bind-mount на хосте никогда не затрагивается). Keystore никогда не очищается командой `rm` — это делает только `coast ssg secrets clear`.
- `coast ssg restart` — это удобная обертка для `stop` + `start`.

## Команды

### `coast ssg run`

Создает DinD `<project>-ssg`, если он не существует, и запускает его внутренние сервисы. Выделяет один динамический порт хоста на каждый объявленный сервис и публикует их на внешнем DinD. Записывает сопоставления в state DB, чтобы allocator портов не использовал их повторно.

```bash
coast ssg run
```

Передает события прогресса через тот же канал `BuildProgressEvent`, что и `coast ssg build`. План по умолчанию состоит из 7 шагов:

1. Подготовка SSG
2. Создание контейнера SSG
3. Запуск контейнера SSG
4. Ожидание внутреннего демона
5. Загрузка кэшированных образов
6. Материализация секретов (без вывода, если нет блока `[secrets]`; иначе выдает элементы по каждому секрету)
7. Запуск внутренних сервисов

**Автозапуск**. `coast run` в потребляющем Coast, который ссылается на сервис SSG, автоматически запускает SSG, если он еще не запущен. Вы всегда можете явно выполнить `coast ssg run`, но обычно это не требуется. См. [Consuming -> Auto-start](CONSUMING.md#auto-start).

### `coast ssg start`

Запускает ранее остановленный SSG. Требует существующего контейнера `<project>-ssg` (то есть ранее должен быть выполнен `coast ssg run`). Повторно материализует секреты из keystore, чтобы любые изменения с момента остановки вступили в силу, а затем заново поднимает host-side checkout socat'ы для всех канонических портов, которые были checked out до остановки.

```bash
coast ssg start
```

### `coast ssg stop`

Останавливает внешний контейнер DinD. Внутренний compose stack останавливается вместе с ним. Контейнер, выделения динамических портов и записи виртуальных портов на уровне проекта сохраняются, чтобы следующий `start` выполнялся быстро.

```bash
coast ssg stop
coast ssg stop --force
```

Host-side checkout socat'ы завершаются, но их записи в state DB сохраняются. Следующий `coast ssg start` или `coast ssg run` запускает их заново. См. [Checkout](CHECKOUT.md).

**Ограничение для удаленных потребителей.** Демон отказывается останавливать SSG, пока его в настоящий момент использует любой удаленный shadow Coast (созданный с помощью `coast assign --remote ...`). Передайте `--force`, чтобы принудительно удалить reverse SSH tunnels и все равно продолжить. См. [Consuming -> Remote Coasts](CONSUMING.md#remote-coasts).

### `coast ssg restart`

Эквивалент `stop` + `start`. Сохраняет контейнер и сопоставления динамических портов.

```bash
coast ssg restart
```

### `coast ssg rm`

Удаляет внешний контейнер DinD. По умолчанию это сохраняет внутренние именованные тома (Postgres WAL и т. д.), поэтому ваши данные переживают циклы `rm` / `run`. Содержимое bind-mount на хосте никогда не затрагивается.

```bash
coast ssg rm                    # сохраняет именованные тома; сохраняет keystore
coast ssg rm --with-data        # также удаляет именованные тома; keystore все равно сохраняется
coast ssg rm --force            # выполняется несмотря на удаленных потребителей
```

- `--with-data` удаляет каждый внутренний именованный том перед удалением самого DinD. Используйте это, когда вам нужна чистая база данных.
- `--force` выполняется даже тогда, когда удаленные shadow Coast'ы ссылаются на SSG. Та же семантика, что и у `stop --force`.
- `rm` очищает строки `ssg_port_checkouts` (разрушительно для host bindings канонических портов).

Keystore — место, где живут нативные для SSG секреты (`coast_image = "ssg:<project>"`) — **не** затрагивается ни `rm`, ни `rm --with-data`. Чтобы стереть секреты SSG, используйте `coast ssg secrets clear` (см. [Secrets](SECRETS.md)).

### `coast ssg ps`

Показывает статус сервисов для SSG текущего проекта. Читает `manifest.json` для собранной конфигурации, затем проверяет live state DB на предмет метаданных запущенных контейнеров.

```bash
coast ssg ps
```

Вывод после успешного `run`:

```text
SSG build: b455787d95cfdeb_20260420061903  (project: cg, running)

  SERVICE              IMAGE                          PORT       STATUS
  postgres             postgres:16                    5432       running
  redis                redis:7-alpine                 6379       running
```

### `coast ssg ports`

Показывает сопоставление canonical / dynamic / virtual портов для каждого сервиса с пометкой `(checked out)`, когда для этого сервиса активен host-side canonical-port socat. Виртуальный порт — это то, к чему на самом деле подключаются потребители. Подробности см. в [Routing](ROUTING.md).

```bash
coast ssg ports

#   SERVICE              CANONICAL       DYNAMIC         VIRTUAL    STATUS
#   postgres             5432            54201           42000      (checked out)
#   redis                6379            54202           42001
```

### `coast ssg logs`

Потоково выводит логи внешнего контейнера DinD или конкретного внутреннего сервиса.

```bash
coast ssg logs --tail 100
coast ssg logs --service postgres --tail 50
coast ssg logs --service postgres --follow
```

- `--service <name>` нацеливает команду на внутренний сервис по compose key; без него вы получаете stdout внешнего DinD.
- `--tail N` ограничивает количество исторических строк (по умолчанию 200).
- `--follow` / `-f` потоково выводит новые строки по мере их появления, до `Ctrl+C`.

### `coast ssg exec`

Выполняет команду внутри внешнего DinD или внутреннего сервиса.

```bash
coast ssg exec -- sh
coast ssg exec --service postgres -- psql -U coast -l
```

- Без `--service` команда выполняется во внешнем контейнере `<project>-ssg`.
- С `--service <name>` команда выполняется внутри соответствующего compose-сервиса через `docker compose exec -T`.
- Всё после `--` передается напрямую в базовый `docker exec`, включая флаги.

### `coast ssg ls`

Перечисляет все SSG, известные демону, во всех проектах. Это единственная команда, которая не определяет проект из cwd; она возвращает строки для каждой записи в состоянии SSG демона.

```bash
coast ssg ls

#   PROJECT     STATUS     BUILD                                       SERVICES   CREATED
#   cg          running    b455787d95cfdeb_20260420061903               2          2026-04-20T06:19:03Z
#   filemap     stopped    b9b93fdb41b21337_20260418123012               3          2026-04-18T12:30:12Z
```

Полезно для обнаружения забытых SSG от старых проектов или для быстрого просмотра, у каких проектов на этой машине есть SSG в каком-либо состоянии.

## Семантика мьютекса

Каждая изменяющая команда SSG (`run`/`start`/`stop`/`restart`/`rm`/`checkout`/`uncheckout`) захватывает мьютекс SSG на уровне проекта внутри демона перед передачей управления реальному обработчику. Два одновременных вызова для одного и того же проекта становятся в очередь; для разных проектов они выполняются параллельно. Команды только для чтения (`ps`/`ports`/`logs`/`exec`/`doctor`/`ls`) не захватывают мьютекс.

## Интеграция с Coastguard

Если вы используете [Coastguard](../concepts_and_terminology/COASTGUARD.md), SPA отображает жизненный цикл SSG на отдельной странице (`/project/<p>/ssg/local`) с вкладками Exec, Ports, Services, Logs, Secrets, Stats, Images и Volumes. `CoastEvent::SsgStarting` и `CoastEvent::SsgStarted` срабатывают всякий раз, когда потребляющий Coast инициирует автозапуск, чтобы UI мог связать запуск с проектом, которому это понадобилось.
