# Переменные

Coastfile поддерживает интерполяцию переменных окружения во всех строковых значениях. Переменные разрешаются во время разбора, до обработки TOML, поэтому они работают в любом разделе и в любой позиции значения.

## Синтаксис

Сошлитесь на переменную окружения с помощью `${VAR_NAME}`:

```toml
[coast]
name = "${PROJECT_NAME}"
compose = "${COMPOSE_PATH}"

[ports]
web = ${WEB_PORT}
```

Имена переменных должны начинаться с буквы или символа подчёркивания, за которыми следуют буквы, цифры или символы подчёркивания (соответствуя шаблону `[A-Za-z_][A-Za-z0-9_]*`).

## Значения по умолчанию

Используйте `${VAR:-default}`, чтобы указать резервное значение, если переменная не установлена:

```toml
[coast]
name = "${PROJECT_NAME:-my-app}"
runtime = "${RUNTIME:-dind}"

[ports]
web = ${WEB_PORT:-3000}
api = ${API_PORT:-8080}
```

Если `PROJECT_NAME` установлена в окружении, используется её значение. Если нет, подставляется `my-app`. Значения по умолчанию могут содержать любые символы, кроме `}`.

## Неопределённые переменные

Когда на переменную ссылаются без значения по умолчанию и она не установлена в окружении, Coast **сохраняет буквальный текст `${VAR}`** и выводит предупреждение:

```
warning: undefined environment variable 'DB_HOST' preserved as literal '${DB_HOST}'; use '${DB_HOST:-}' for explicit empty, or '$${DB_HOST}' to escape entirely
```

Сохранение ссылки (вместо тихой замены её пустой строкой) позволяет shell-командам вроде `ARCH=$(uname -m) && curl .../linux-${ARCH}.tar.gz` продолжать работать — shell в Dockerfile всё ещё сможет раскрыть `${ARCH}` во время сборки, даже если Coast никогда её не устанавливал.

Если вы действительно хотите пустую подстановку, когда переменная отсутствует, используйте явное пустое значение по умолчанию:

```toml
[coast]
name = "${PROJECT_NAME:-}"   # "" когда PROJECT_NAME не установлена
```

Если вам нужен буквальный текст `${VAR}` без каких-либо предупреждений, экранируйте его с помощью `$${VAR}` (см. [Экранирование](#escaping) ниже).

## Экранирование

Чтобы получить буквальный `${...}` в вашем Coastfile (например, в значении, которое должно содержать текст `${VAR}`, а не его раскрытое значение), удвойте начальный знак доллара:

```toml
[coast.setup]
run = ["echo '$${NOT_EXPANDED}'"]
```

Это создаст буквальную строку `echo '${NOT_EXPANDED}'` без попытки поиска переменной.

## Примеры

### Секреты с ключами из переменных окружения

```toml
[secrets.api_key]
extractor = "env"
var = "${API_KEY_ENV_VAR:-MY_API_KEY}"
inject = "env:API_KEY"
```

### Общая конфигурация сервисов

```toml
[shared_services.postgres]
image = "postgres:${PG_VERSION:-16}"
env = [
    "POSTGRES_USER=${DB_USER:-coast}",
    "POSTGRES_PASSWORD=${DB_PASSWORD:-dev}",
    "POSTGRES_DB=${DB_NAME:-coast_dev}",
]
ports = [5432]
```

### Путь к compose для каждого окружения

```toml
[coast]
name = "my-app"
compose = "${COMPOSE_FILE:-./docker-compose.yml}"
```

## Переменные и секреты

Интерполяция переменных и [секреты](SECRETS.md) служат разным целям:

| | Переменные (`${VAR}`) | Секреты (`[secrets.*]`) |
|---|---|---|
| **Когда разрешаются** | Во время разбора (до обработки TOML) | Во время сборки (извлекаются из настроенных источников) |
| **Где хранятся** | Встраиваются в разрешённый Coastfile | Зашифрованное хранилище ключей (`~/.coast/keystore.db`) |
| **Сценарий использования** | Конфигурация, которая различается между окружениями (порты, пути, теги образов) | Чувствительные учётные данные (API-ключи, токены, пароли) |
| **Видны в артефактах** | Да (значения появляются в `coastfile.toml` внутри сборки) | Нет (в манифесте появляются только имена секретов) |

Используйте переменные для неконфиденциальной конфигурации, которая меняется между машинами или CI-окружениями. Используйте секреты для значений, которые никогда не должны появляться в артефактах сборки.
